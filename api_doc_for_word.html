📜 DevSphere Tech Architecture & API Docs

This technical guide provides full schema definitions and payload examples for the DevSphere Leaderboard system integration.

🛰️ 1. External Webhook API

Used by GitHub Actions or automation scripts to grant points to participants.

Endpoint: POST /api/webhooks/github

Payload Key Data Type Req? Description / Example
sender.login String YES GitHub username from the sender. (e.g., octocat)
repository.full_name String YES Organization/Repo identifier for task mapping.
task_title String YES Category of the task (e.g., web, ml, foss).
difficulty String YES Task difficulty level (e.g., easy, medium, hard).
commit_time String YES ISO-8601 timestamp of the commit (for calculating exact time taken).

📦 Sample JSON Payload:

{
  "sender": {
    "login": "k"
  },
  "repository": {
    "full_name": "MyOrgs/MachineLearningChallenge"
  },
  "task_title": "ml",
  "difficulty": "hard",
  "commit_time": "2026-04-04T15:25:00+05:30"
}

📦 Success Response (200 OK):

{
  "message": "Task completion recorded successfully"
}

🗄️ 2. Database Schema Definitions

The following tables represent the core data structures in Supabase.

Table: profiles (Participants)

Column Type Constraint Purpose
idUUIDPrimary KeySupabase Auth mapping.
emailTextUniqueVerified college email.
github_usernameTextUniqueUnique identifier for rankings.
scoreBigIntDefault 0Aggregated total points.
time_takenBigInt-Time in seconds since event start.

Table: tasks (Event Metadata)

Column Type Constraint Purpose
idUUIDPrimary KeyInternal task ID.
github_identifierTextUniqueMatch for repository.full_name.
titleText-Task Category (Web, ML, etc).
difficultyText-Level (Easy, Med, Hard).
pointsInteger-Reward amount for completion.

Table: task_completions (Transaction Log)

Column Type Constraint Purpose
idUUIDPrimary KeyTransaction ID.
profile_idUUIDFK (profiles)Who completed it.
task_idUUIDFK (tasks)Which task was completed.
statusEnum'valid' / 'invalid'Admin toggle for validity.
payloadJSONB-Raw backup of the webhook data.

🔄 3. Internal Application Payloads

Onboarding Flow (Participant Setup)

Triggered when a user first enters their GitHub id on /onboarding.

PATCH /rest/v1/profiles?id=eq.{user_id}
{
  "github_username": "fcuked",
  "id": "e8a2..."
}

Admin Submission Validation

Admin hub action to manually invalidate or validate a result.

PATCH /rest/v1/task_completions?id=eq.{submission_id}
{
  "status": "invalid"
}
